# 商材・業務種別の二重化 解消作業（engagement merge）

最終更新: 2026-06-10 / 担当: Claude (Fable 5) + むー様

---

## 0. これは何の作業か（背景）

営業代行の商材レイヤー（products / business_categories / engagements）導入時に、
**「M&A」などの商材（business_category）が同じものを2個ずつ作ってしまった**。
その結果、配下の業務種別（売り手ソーシング / 買い手マッチング / クライアント開拓 など）も
2個ずつ存在し、日々の架電リスト・アポ・クライアントが**2つの箱にバラバラに入っている**。

今は営業代行タブが「2つの箱をまとめて表示」しているため表面化していないが、
**買い手マッチングを独立画面・クライアントポータルに切り出すと、片方の箱のデータが
画面に出ない／集計がズレる**という事故が起きる。本格化の前に1個に統合する。

```
本来あるべき姿                    今の実際
営業代行                          営業代行
 └ M&A（1つ）                      ├ M&A（箱①）  ← データ分散
     ├ 売り手ソーシング            └ M&A（箱②・コピー） ← データ分散
     ├ 買い手マッチング
     └ クライアント開拓
```

## 5. 安全策

- **データは消さない**。主役データ(アポ/リスト/クライアント)は付け替えUPDATEのみ。
  削除するのは「寄せ先に既に同じものがある重複設定行」だけ（情報損失なし）。
- **画面の見た目・操作は変わらない**（slugは同じ。archiveされた重複が裏で消えるだけ）。
- apply_migration は単一トランザクション → 途中でエラーが出れば全て巻き戻る（中途半端な状態にならない）。
- [[feedback_rls_changes]] 大規模変更は低トラフィック時に / [[feedback_db_no_drop]] tmp_*はDROPしない。
