# スパキャリ 事後課題 生成エンジン 再設計 設計書（v1ドラフト）

作成: 2026-06-15 / 対象: 第2回以降のAI事後課題生成（第1回STEP1-7共通テンプレは別管理）

## 0. 背景・現状
- 現状 `generate-spacareer-homework30`（想定モデル Claude Haiku 4.5）は**本番未デプロイ**で、コードは mock 生成にフォールバック中。実質AI逆算は未稼働。
- 本再設計で「目標から逆算した具体アクションプランを成果物として可視化させる」本実装を行う。
- **使用モデル: Claude Sonnet 4.6**（むー様決定 2026-06-15）。

## 1. 目的（むー様の意図）
受講生を「課題をこなす生徒」ではなく「自分の事業のCEO」と位置づけ、AIは月額で雇える従業員。
目標から逆算した実行可能なアクションプランを、数十時間の泥臭い実働が出る粒度まで分解し、
数値基準の成果物で「やったふり」を排除する。

## 2. 4階層分解モデル（KGI → KPI → KDI → 日次/週次タスク）
- **KGI**: 最終目標（例: 〇ヶ月後に月商◯◯万）。基本情報の目標＋議事録から確定。
- **KPI**: KGI達成に必要な中間指標（受注件数・商談数・必要リード数 等）を逆算。
- **KDI（行動指標）**: KPIを生む行動量（テレアポ件数・フォーム送信数・SNS投稿数 等）を逆算。
- **日次/週次タスク**: KDIを稼働日に配分した実行タスク。ここまで割り切ると自然に50〜60項目になる。
  - ※「50〜60項目」は**目的でなく結果**。分解しきった結果としての項目数。

### 逆算の例（仕様確認用・値は例）
月20万（法人営業）→ 1件5万なら受注4件 → 成約率10%で商談40件 → アポ率0.5%でテレアポ8,000件
→ 週8,000件は非現実 → フォーム営業0.01%なら送信◯件 等、実行可能な手段・配分へ変換。

## 3. 仮定（assumption）の扱い
- 成約率・アポ率・フォーム反応率など**仮置きした全数値に「仮定」タグ**を付与（item に `is_assumption: true` と `assumption_basis`）。
- **仮定値を検証する課題自体を課題リストに含める**（例: 「初週のアポ率を実測し、想定0.5%との差を記録」）。
- トレーナーが毎回確認するため、仮定は隠さず明示する。

## 4. 実行可能性の自動調整（容量フィット）
- 基準値 = **キックオフヒアリングQ24「スパキャリのために確保できる週あたりの時間（自己宣言）」**（補助: Q22平日/Q23休日の自由時間）。
- 各タスクに `est_minutes`（所要見積）を持たせ、
  **総見積時間が「週稼働可能時間 × 期間」の 100〜130% に収まるよう自動調整**。
  - 超過時: 手段転換（テレアポ→フォーム）、日次配分の平準化、優先度の低いタスクの圧縮で調整。
  - 不足時: 深掘りタスク/検証タスクを追加して密度を上げる。
- 自己効力感維持のため、物理的に不可能な総量は出さない（130%上限）。

## 5. 数値基準の成果物（やったふり防止）
- 各課題に**検証可能な提出フォーマットと数値合格基準**を必須化。
  - 例: 「テレアポする」ではなく「コール記録シートに**150件分**のログ（日時・社名・結果・トーク）が記入済み」。
- item フィールド: `deliverable_type`（text/file/spreadsheet/image）、`deliverable_spec`（必須列・行数等）、`numeric_threshold`（例: 150）。
- SNS例: 単なる投稿でなく「過去投稿のバズ比較分析レポート（指標列付き）」を提出物に。

## 6. CEO意思決定 / AI委任 の分離
- 各item に `owner`: `ceo`（受講生自身の意思決定・判断）／`ai`（AIに委任して実行/下書きさせる作業）。
- 課題構造を「CEOとして決めること」と「従業員（AI）に任せること」で明確に区分して提示。

## 7. 継続率（multi-month）の織り込み
- KGIが複数月（例: 2ヶ月後30万、継続率50%）の場合、
  既存売上の継続分と新規必要分を分離し、**継続施策（いつ・何をするか）**まで具体タスク化。

## 8. 商材別テンプレ（チャネル）
- テレアポ／フォーム営業／SNS(X)等、手段別の標準分解テンプレを用意。
  - SNS-X例: 商材リサーチ → アカウント作成 → 1日◯ポスト → 投稿方針の具体化 → バズ分析レポート。

## 9. 出力スキーマ（ドラフト）
```jsonc
{
  "kgi": { "statement": "...", "target_value": 200000, "unit": "円/月", "deadline": "YYYY-MM-DD" },
  "layers": [
    {
      "level": "KPI" | "KDI" | "task",
      "title": "...",
      "metric": { "name": "商談数", "value": 40, "unit": "件" },
      "is_assumption": false,
      "assumption_basis": null,            // is_assumption=true の根拠/相場メモ
      "owner": "ceo" | "ai",
      "channel": "telephone" | "form" | "sns_x" | null,
      "est_minutes": 90,
      "deliverable_type": "spreadsheet",
      "deliverable_spec": "列: 日時/社名/結果/トーク。150行以上",
      "numeric_threshold": 150,
      "schedule": { "week": 1, "day": "Mon", "qty": 30 },
      "parent_ref": "KPI-1"
    }
  ],
  "capacity": { "weekly_hours": 15, "period_weeks": 4, "fit_ratio": 1.12 },
  "assumptions_to_validate": [ "アポ率0.5%（初週実測で検証）" ]
}
```
- 既存 `spacareer_homework_items` への落とし込み: position / question_text / question_hint / is_required /
  item_type(file時) / template_url 等にマップ。追加メタ（owner, is_assumption, est_minutes, numeric_threshold等）は
  新カラムまたは items の JSONB メタ列を追加して保持（要設計判断）。

## 10. アーキテクチャ
- Edge Function `generate-spacareer-homework30` を**本実装・デプロイ**（Sonnet 4.6）。
  - 入力: customer_id, next_session_no, 目標（基本情報）, 議事録（前回 minutes_final/draft）,
    キックオフヒアリング回答（Q24等）, 強み診断（第2回以降）。
  - 出力: 上記スキーマ → トレーナー確認後に確定（既存の生成→手動修正→確定フローを踏襲）。
- プロンプトは `spacareer_templates(template_type='ai_prompt')` 管理（運営のみ編集可）に集約。
